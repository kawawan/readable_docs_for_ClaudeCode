# 追加開発 計画：世代間引きエンジン ＋ 容量ウォッチャ ＋ ドライラン

> 状態の正＝`追加開発/TODO.md`／改訂の経緯＝`作業メモ/作業ログ.md`（append-only・既存へ追記）／設計根拠＝本ファイル。
> 位置づけ：初期プロト（ルートの `計画.md`／`TODO.md`＝取得側の実装・Step4でクローズ）に続く **追加開発**。
> 進行スキルは `/atticbak-addon`（0→1用の `/atticbak-prototype` とは別）。

## Context（なぜやるか）

- 本来目的は **自宅の複数マシン（母艦NAS・書斎機・親のノート）のバックアップを1本のツールで世代管理する**こと。取るだけなら既存の rsync で足りているが、**古い世代を賢く間引く**部分が手作業で破綻しかけている。
- 現状のプロトは **取得専用**（各マシンから母艦へ丸ごとコピーするだけ・消す仕組みが無い）。結果、母艦の空きが月末に必ず枯渇し、そのたび手で `rm` している。事故のもと。**保持ポリシーに基づく自動間引き** を組み込む（バックログ P2）。
- 現状は **世代の概念が無い**（同名ディレクトリを上書きコピーしていて、3日前の状態に戻せない）。「いつ時点のスナップショットが今どれだけ残っているか」を機械が把握できない（バックログ Q1）。ロールバックの土台も無い。
- ゴール：**世代間引きエンジン（GFS方式）＋ 容量ウォッチャ（逼迫検知）＋ ドライラン（消す前に必ず確認）** を作り、将来の「マシン横断の重複排除」の土台を用意する。

保存先の実容量に非依存で先行できる部分（フェーズA＝間引きロジック・ドライラン・世代タグ付け）と、実際の逼迫閾値チューニングに依存する部分（フェーズB＝容量発火の自動運用）に分ける。

## 確定した設計判断（対話で合意・2026-06-22）

- **間引きは GFS（Grandfather-Father-Son）**。日次7・週次4・月次12 を既定とし、どの世代がどの枠を埋めているかは **世代タグ** で表現する。単純な「N日より古いものを消す」方式は取らない（週次・月次の代表世代が巻き添えで消えるため）。
- **不可逆操作の不変条件＝「エンジンは削除候補の選定だけ、実削除は別レイヤ」**。選定ロジックと `unlink` の実行を必ず分離する。ドライランはこの分離があって初めて成立する。
- **`守護世代` は絶対に間引かない**。手動で「これは残す」と印を付けたスナップショットは、GFS の計算対象から最初に除外する。誤消去の最終防波堤。
- **世代の識別はスナップショット取得完了時刻（`YYYYMMDD-HHMMSS`）**。ディレクトリ名そのものをIDとして扱い、別途の採番テーブルは持たない（ファイルシステムを正とする思想）。
- **間引き判定の入力は「実ファイル走査」ではなく `世代マニフェスト`（JSON）**。毎回全ディレクトリを stat すると遅いので、取得時にマニフェストへ1行追記し、エンジンはそれだけ読む。
- **容量逼迫の検知は「使用率しきい値」で設計・動作確認**（モック値）。実際の発火閾値（80%か90%か）は実機の増加ペースを見てから決める＝運用確認事項。

## データ構造（新設）

保存先：本番は母艦NASの `/pool/backup/`（新規）。プロト検証では作業機の `~/lab/atticbak-sandbox/` に作る。

### 世代マニフェスト（`manifest.jsonl`・追記オンリー）
取得のたびに1行追記する台帳。DBは使わず JSON Lines。
- `snapshot_id`（取得完了時刻 `YYYYMMDD-HHMMSS`）
- `source`（取得元マシン名：`nas` / `study` / `parents`）
- `size_bytes`（そのスナップショットの実バイト数）
- `generation_tag`（`daily` / `weekly` / `monthly`・間引きエンジンが後付けで更新）
- `guarded`（守護世代フラグ・既定 false）
- 役割：間引きエンジンの唯一の入力 ＋ 容量ウォッチャの集計元。実ファイルは走査しない。

### 削除計画（`prune_plan.json`・その都度生成・使い捨て）
- エンジンが「今回消す候補」を書き出す中間ファイル。
- `to_delete`（snapshot_id の配列）／`to_keep_reason`（残す世代とその理由：`daily枠` `guarded` 等）／`freed_bytes_estimate`。
- ドライランはこのファイルを人間に見せて終わる。実削除レイヤはこのファイルだけを入力に取る（エンジンの再実行はしない＝計画と実行の乖離を防ぐ）。

### 既存の扱い
- `rsync` の取得スクリプト（プロトで実装済み）… 変更なし。末尾にマニフェスト追記の1行を足すだけ。
- `latest/` シンボリックリンク（最新世代を指す）… 間引き対象外。エンジンは snapshot 実体だけ見る。

## 間引きエンジン（フェーズA・コア）

`atticbak/engine.py`。UIは無い。以下を直列で通す純粋関数の並び：
`manifest.load` → `classify.assign_generation_tags`（GFS枠へ割当）→ `guard.exclude_guarded`（守護世代を除外）→ `select.pick_prune_candidates`（枠から溢れた世代を候補化）→ `plan.write_prune_plan`（`prune_plan.json` 出力）。**ここまでで削除は一切しない**。

- **入力**：`manifest.jsonl` のみ。走査コスト0。
- **GFS割当**：新しい順に日次枠7を埋め、溢れたもののうち週境界の代表を週次枠4へ、さらに溢れたものの月境界代表を月次枠12へ。どの枠にも入らなかった世代が削除候補。
- **守護世代**：`guarded=true` は枠計算の前に退避。枠を消費しない（守護があっても通常世代の保持数は減らさない）。
- **出力**：削除候補と「なぜ残すか」を明記した `prune_plan.json`。人間が読める理由付き。

## ドライラン（フェーズA・明示工程）

削除を伴わない確認モードを既定にする。`--apply` を付けたときだけ実削除に進む：
- `atticbak prune`（引数なし）… `prune_plan.json` を生成し、標準出力へ「消す候補・残す理由・解放見込み容量」を表形式で出すだけ。
- `atticbak prune --apply` … 既存の `prune_plan.json` を読み、実削除レイヤへ渡す。**計画生成と実削除は必ず別プロセス**（同一実行で計画→即削除はしない）。
- 計画が古い（生成後にマシンが新しく取得した）場合は `--apply` を弾く。マニフェストの最終行タイムスタンプと計画の生成時刻を照合。

## 危険操作の設計（フェーズA・明示工程）

- **選定ミス（残すべきを消す）**：GFS計算とは独立に守護世代を最優先除外。加えて「1回の間引きで総世代数の半分以上を消す計画」は異常とみなし、`--apply` 前に強制停止して警告。
- **実行時例外**：`unlink` 途中での失敗はログへ記録し、途中まで消した分と残りをマニフェストへ反映してから停止（マニフェストと実体の乖離を残さない）。
- **判定順**：異常検知（半数超え削除・守護世代が候補に混入）を**できるだけ早く**（計画生成直後）＝実削除の**前**で止める。
- **原子性**：1世代の削除は「実体削除 → マニフェストの該当行を tombstone 化」をペアで行う。片方だけ成功した状態を残さない。
- **削除失敗**：最低限ログ・無限リトライ禁止・次回実行時に未完了を検知して再開。

## 容量ウォッチャ（フェーズA・逼迫検知）

- **監視層**：`df` 相当で母艦の使用率を取得し、しきい値（モックで80%）を超えたら「間引き推奨」フラグを立てるだけ（この段階では自動削除しない）。
- **集計層**：`manifest.jsonl` の `size_bytes` を source 別・generation_tag 別に合計し、「どのマシンのどの世代が容量を食っているか」を出す。
- **通知**：逼迫時に自宅Discordの個人チャンネルへ「そろそろ間引き時です・解放見込みXX GB」を投げる。実削除は人間が `--apply` で判断。

## 自動運用（フェーズB・追加開発・発火閾値は想定値）

- 自動運用 ＝ **容量ウォッチャが逼迫を検知したら、間引き計画の生成までを cron で自動化し、`--apply` だけ人間に残す**。
- **発火閾値は与えられた設定値として扱う体**で設計・動作確認（モック値）。実際の閾値は実機の増加ペースを観測してから確定（運用確認）。
- 常駐ではなく systemd timer（or cron）駆動。現在の取得スクリプトは手動起動なので **timer 定義の追加とログローテーション** が必要。

## 未確定・運用確認事項

- **発火閾値**：使用率80%で警告か90%か（＝容量ウォッチャの感度）。実機の月間増加ペース次第。
- **GFS の既定値の妥当性**（日次7・週次4・月次12。母艦の実容量に対して過剰/過少かは運用で見る）。
- **親のノート（`parents`）の扱い**（取得頻度が不定期。日次枠の前提が崩れる可能性）。
- **マニフェストの肥大化対策**（tombstone 行が溜まった際のコンパクション・バックログ Q4）。
- **半数超え削除ガードのしきい値**（半分が妥当か、別の比率か。危険操作設計の成果として確定）。

## Verification（動作確認の筋）

サンドボックスの `~/lab/atticbak-sandbox/` にダミー世代を仕込んで実機確認：
- `atticbak prune`（ドライラン）で削除候補・残す理由・解放見込みが表示され、**実体は1つも消えない**。
- GFS割当：日次7・週次4・月次12の枠に沿って残り、溢れた中間世代だけが候補化される。
- 守護世代ガード：`guarded=true` を付けた世代が、枠から溢れる位置にあっても候補に入らない。
- 半数超え削除ガード：意図的に大量の削除候補を作り、`--apply` が強制停止することを確認。
- 原子性：`--apply` の途中を kill しても、マニフェストと実体の食い違いが残らない（tombstone で追跡できる）。
- 容量ウォッチャ：モック使用率をしきい値超えに設定し、Discord通知が飛ぶ。
- （フェーズB）cron/timer 駆動で計画生成まで自動化され、`--apply` だけ手動に残ることを確認。
