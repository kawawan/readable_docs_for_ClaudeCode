# 追加開発の計画：世代間引きエンジン・容量ウォッチャ・ドライラン

## Context（なぜやるか）

本来の目的は、自宅の複数マシン（母艦NAS・書斎機・親のノート）のバックアップを、1本のツールで世代管理することです。

取得そのものは、既存の rsync で足りています。破綻しかけているのは、古い世代を賢く間引く部分です。ここがいまは手作業になっています。

現状のプロトは取得専用で、消す仕組みがありません。各マシンから母艦へ丸ごとコピーするだけです。そのため母艦の空きが月末に必ず尽き、そのたびに手で `rm` しています。事故のもとです。

世代の概念もありません。同名ディレクトリを上書きコピーしているので、3日前の状態に戻せません。どの時点のスナップショットが今どれだけ残っているかを、機械が把握できません。ロールバックの土台もありません。

ゴールは次の3つを作ることです。将来の「マシン横断の重複排除」の土台にもします。

- 世代間引きエンジン（GFS方式）

- 容量ウォッチャ（逼迫の検知）

- ドライラン（消す前に必ず確認する）

作業は2つのフェーズに分けます。分ける基準は、保存先の実容量に依存するかどうかです。

| フェーズ | 中身 | 実容量への依存 |
| --- | --- | --- |
| A | 間引きロジック・ドライラン・世代タグ付け | 依存しない。先行できる |
| B | 容量発火の自動運用 | 逼迫閾値のチューニングに依存する |

## 確定した設計判断（対話で合意・2026-06-22）

**間引きは GFS（Grandfather-Father-Son）方式にする。**
日次7・週次4・月次12 を既定とする。どの世代がどの枠を埋めているかは、世代タグで表す。単純な「N日より古いものを消す」方式は取らない。週次・月次の代表世代まで巻き添えで消えてしまうからだ。

**選定と実削除を分ける。これを不変条件とする。**
エンジンは削除候補を選ぶだけにして、実際の `unlink` は別レイヤに置く。ドライランは、この分離があって初めて成り立つ。

**守護世代は絶対に間引かない。**
手動で「これは残す」と印を付けたスナップショットは、GFS の計算に入れる前に除外する。誤消去の最終防波堤である。

**世代の識別には取得完了時刻（`YYYYMMDD-HHMMSS`）を使う。**
ディレクトリ名そのものを ID として扱い、別の採番テーブルは持たない。ファイルシステムを正とする考え方だ。

**間引き判定の入力は、世代マニフェスト（JSON）にする。**
実ファイルは走査しない。毎回すべてのディレクトリを stat すると遅いので、取得時にマニフェストへ1行追記し、エンジンはそれだけを読む。

**容量逼迫は、使用率のしきい値で検知する。**
設計と動作確認はモック値で行う。実際の発火閾値を80%にするか90%にするかは、実機の増加ペースを見てから決める。これは運用確認事項とする。

## データ構造（新設）

保存先は、本番が母艦NASの `/pool/backup/`（新規）です。プロト検証では、作業機の `~/lab/atticbak-sandbox/` に作ります。

### 世代マニフェスト（`manifest.jsonl`・追記オンリー）

取得のたびに1行追記する台帳です。DBは使わず JSON Lines にします。

| フィールド | 意味 |
| --- | --- |
| `snapshot_id` | 取得完了時刻 `YYYYMMDD-HHMMSS` |
| `source` | 取得元マシン名（`nas` / `study` / `parents`） |
| `size_bytes` | そのスナップショットの実バイト数 |
| `generation_tag` | `daily` / `weekly` / `monthly`。間引きエンジンが後付けで更新する |
| `guarded` | 守護世代フラグ。既定は false |

役割は2つです。間引きエンジンの唯一の入力であり、容量ウォッチャの集計元でもあります。実ファイルは走査しません。

### 削除計画（`prune_plan.json`・その都度生成・使い捨て）

エンジンが「今回消す候補」を書き出す中間ファイルです。

| フィールド | 意味 |
| --- | --- |
| `to_delete` | 消す `snapshot_id` の配列 |
| `to_keep_reason` | 残す世代とその理由（`daily枠` `guarded` など） |
| `freed_bytes_estimate` | 解放できる見込みのバイト数 |

ドライランは、このファイルを人間に見せて終わります。実削除レイヤは、このファイルだけを入力に取ります。エンジンの再実行はしません。計画と実行の乖離を防ぐためです。

### 既存の扱い

- `rsync` の取得スクリプト（プロトで実装済み）は変更しない。末尾にマニフェスト追記の1行を足すだけにする。

- `latest/` シンボリックリンク（最新世代を指す）は間引き対象外にする。エンジンはスナップショット実体だけを見る。

## 間引きエンジン（フェーズA・コア）

ファイルは `atticbak/engine.py` です。UI はありません。純粋関数を直列に並べます。前段から順に次のとおりです。

1. **`manifest.load`** — マニフェストを読む。

2. **`classify.assign_generation_tags`** — GFS の枠へ世代を割り当てる。

3. **`guard.exclude_guarded`** — 守護世代を除外する。

4. **`select.pick_prune_candidates`** — 枠から溢れた世代を候補にする。

5. **`plan.write_prune_plan`** — `prune_plan.json` を出力する。

ここまでで削除は一切しません。各部の要点は次のとおりです。

- 入力は `manifest.jsonl` のみ。走査コストはゼロ。

- GFS 割当は、新しい順に日次枠7を埋める。溢れたもののうち週境界の代表を週次枠4へ回す。さらに溢れたものの月境界代表を月次枠12へ回す。どの枠にも入らなかった世代が削除候補になる。

- 守護世代（`guarded=true`）は枠計算の前に退避する。枠を消費しない。守護世代があっても、通常世代の保持数は減らさない。

- 出力は、削除候補と「なぜ残すか」を明記した `prune_plan.json`。人間が読める理由付きにする。

## ドライラン（フェーズA・明示工程）

削除を伴わない確認モードを既定にします。`--apply` を付けたときだけ、実削除に進みます。

- `atticbak prune`（引数なし）は、`prune_plan.json` を生成する。標準出力へ「消す候補・残す理由・解放見込み容量」を表で出すだけにする。

- `atticbak prune --apply` は、既存の `prune_plan.json` を読み、実削除レイヤへ渡す。計画生成と実削除は必ず別プロセスにする。同じ実行で「計画してすぐ削除」はしない。

- 計画が古いときは `--apply` を弾く。生成後にマシンが新しく取得した場合が該当する。マニフェストの最終行のタイムスタンプと、計画の生成時刻を照合する。

## 危険操作の設計（フェーズA・明示工程）

- 選定ミス（残すべきものを消す）に備える。GFS 計算とは独立に、守護世代を最優先で除外する。加えて「1回の間引きで総世代数の半分以上を消す計画」は異常とみなし、`--apply` の前に強制停止して警告する。

- 実行時例外に備える。`unlink` の途中で失敗したら、ログに記録する。途中まで消した分と残りをマニフェストへ反映してから停止する。マニフェストと実体の乖離を残さない。

- 判定順を決める。異常検知（半数超えの削除、守護世代の候補混入）は、できるだけ早く止める。計画生成の直後、実削除の前で止める。

- 原子性を守る。1世代の削除は「実体の削除」と「マニフェスト該当行の tombstone 化」をペアで行う。片方だけ成功した状態を残さない。

- 削除失敗に備える。最低限ログを残す。無限リトライはしない。次回実行時に未完了を検知して再開する。

## 容量ウォッチャ（フェーズA・逼迫検知）

- 監視層は、`df` 相当で母艦の使用率を取る。しきい値（モックで80%）を超えたら「間引き推奨」フラグを立てるだけにする。この段階では自動削除しない。

- 集計層は、`manifest.jsonl` の `size_bytes` を source 別・generation_tag 別に合計する。どのマシンのどの世代が容量を食っているかを出す。

- 通知は、逼迫時に自宅 Discord の個人チャンネルへ投げる。「そろそろ間引き時です・解放見込み XX GB」と伝える。実削除は人間が `--apply` で判断する。

## 自動運用（フェーズB・追加開発・発火閾値は想定値）

- 自動運用とは、容量ウォッチャが逼迫を検知したら、間引き計画の生成までを cron で自動化することを指す。`--apply` だけは人間に残す。

- 発火閾値は、与えられた設定値として扱う体で設計・動作確認する（モック値）。実際の閾値は、実機の増加ペースを観測してから確定する（運用確認）。

- 常駐ではなく systemd timer（または cron）で駆動する。現在の取得スクリプトは手動起動なので、timer 定義の追加とログローテーションが要る。

## 進め方（作業順序）

大掛かりなので次の順で進めます。各ステップの完了ごとに、永続層（本TODO・計画・作業ログ）へ記録します。実行はセッション（週）を跨ぐ前提です。

1. **計画・TODO 整備＋スキル新設**（このセッションで完了）。本計画・TODO を新規作成する。作業ログへキックオフを追記する。`/atticbak-addon` スキルを新設する。

2. **危険操作の設計・異常検知ルール**。「消してはいけない条件」をカタログ化する。守護世代の扱い、半数超え削除ガード、原子性、判定順を確定する。以降のエンジン実装がここに依存するので、先に固める。

3. **ドキュメント類の作成**。DOC-02（世代管理ロジック＝GFS）、DOC-01（マニフェスト／削除計画のスキーマ）、DOC-04（非機能＝実行権限・sudo境界）、DOC-05（運用＝危険操作・ドライラン）を更新する。気づき・改善案とバックログも更新する。

4. **サンドボックス整備**。`~/lab/atticbak-sandbox/` に、ダミー世代（日付違いの空ディレクトリ群）とサンプル `manifest.jsonl` を生成するスクリプトを作る。

5. **実装**。`engine.py`（GFS割当から計画出力まで）、`apply.py`（実削除レイヤ・計画のみを入力）、容量ウォッチャの結線を作る。そのあと Verification に進む。

上記の1〜5がフェーズAです。cron 自動運用と timer 定義はフェーズBで、A が固まってからの追加開発とします（発火閾値は想定値でモック確認）。

## Verification（動作確認の筋）

サンドボックス `~/lab/atticbak-sandbox/` にダミー世代を仕込んで、実機で確認します。

- ドライラン：`atticbak prune` で削除候補・残す理由・解放見込みが表示され、実体は1つも消えない。

- GFS 割当：日次7・週次4・月次12 の枠に沿って残り、溢れた中間世代だけが候補になる。

- 守護世代ガード：`guarded=true` を付けた世代が、枠から溢れる位置にあっても候補に入らない。

- 半数超え削除ガード：意図的に大量の削除候補を作り、`--apply` が強制停止することを確認する。

- 原子性：`--apply` の途中を kill しても、マニフェストと実体の食い違いが残らない（tombstone で追跡できる）。

- 容量ウォッチャ：モック使用率をしきい値超えに設定し、Discord 通知が飛ぶ。

- （フェーズB）cron／timer 駆動で計画生成まで自動化され、`--apply` だけ手動に残ることを確認する。
